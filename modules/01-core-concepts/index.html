0 先理解：「Agent」到底是什么？
如果你已经理解 Agent 概念，可以直接跳到第 1 节

在深入 Hermes 之前，我们需要先对齐一个基础概念：什么叫 AI Agent？

🧍 类比：员工 vs 搜索引擎

🔍 传统 AI（ChatGPT 等）

像一个知识渊博的顾问——你问一个问题，它给你一个答案。然后对话结束。

  • 不会主动做事
  • 不会记住你上次说了什么
  • 无法操作你的电脑或系统

🤖 AI Agent（Hermes 等）

像一个能干的员工——你交代一个任务，它自己规划、执行、检查结果，过程中可能用各种工具。

  • 能主动调用工具（终端、文件、浏览器…）
  • 跨会话记住你的偏好和历史
  • 可以无人值守定时运行

🧠 Agent 的四个核心能力

所有 AI Agent（不限于 Hermes）都具备这四个维度的能力：

1️⃣ 感知 (Perceive)

读取输入——用户消息、文件内容、API 返回、终端输出。Agent 需要"看到"发生了什么。

2️⃣ 推理 (Reason)

理解意图、拆解任务、决定下一步做什么。这是 LLM（大语言模型）的核心价值。

3️⃣ 行动 (Act)

实际执行——写文件、跑命令、调 API、发消息。Agent 不只是"想"，还要"做"。

4️⃣ 记忆 (Remember)

把重要信息存下来，下次还能用。没有记忆的 Agent 每次都是"第一次见面"。

🤔 停下来想一想：ChatGPT 有这四种能力吗？它有感知（读你的消息）和推理（生成回答），但没有行动能力（不能操作你的系统）和持久记忆（关了对话就忘）。这就是 Agent 和 Chatbot 的本质区别。

1 Hermes Agent 的核心定位
它不是什么？它是什么？它为什么这样设计？
❌ 常见误解：Hermes = ChatGPT + 几个插件
✅ 正确理解：Hermes 是一个有记忆、会学习、能独立干活的自主 Agent 进程。它不是「你问一句它答一句」的 chatbot——它是「你交代一个目标，它自己去完成」的数字员工。

🏗️ Hermes 的四大设计理念

🔓 Provider 无关

不绑定任何一家模型厂商。支持 OpenRouter、Anthropic、OpenAI、DeepSeek、本地模型等 20+ Provider。

你可以在任务中途切换模型，不影响上下文。

🧠 自我进化

Agent 完成一个复杂任务后，可以自动保存为 Skill。下次遇到类似任务时自动加载，越用越强。

这是 Hermes 最独特的差异化特性。

🌐 无处不在

同一套 Agent 逻辑跑在终端、飞书、Telegram、Discord 等 10+ 平台。一次配置，处处可达。

Gateway 层负责平台适配，Agent 核心逻辑不变。

⏰ 自主运行

通过 Cron 定时任务，Agent 可以在没有人操作的情况下自动完成工作。

比如：每天早上 9 点自动汇总昨天的 Git 提交并推送到群里。

📌 本节要点
  • Hermes 不是 Chatbot 套壳，是自主 Agent 框架
  • 四大支柱：Provider 无关、自我进化、多平台、自动化
  • 同类产品中最独特的差异化：Skills 自学习系统
2 核心引擎：Agent Loop（循环）
这是整个框架的心跳——理解了这个循环，就理解了 Hermes 怎么"想"和"做"

🎬 先看一个具体例子

👤

用户说：

"帮我在 ~/projects/myapp 目录下创建一个 Python 项目，用 FastAPI，带一个 /health 端点。"

下面是 Hermes 实际的处理过程——每一步都是一个 Turn 内的子步骤：

步LLM 返回什么Agent 做什么结果
1Tool Call: terminal("mkdir -p ~/projects/myapp") 执行终端命令 目录创建成功
2Tool Call: write_file("~/projects/myapp/main.py", ...) 写入 FastAPI 代码 文件已写入
3Tool Call: write_file("~/projects/myapp/requirements.txt", ...) 写入依赖文件 文件已写入
4Tool Call: terminal("cd ~/projects/myapp && pip install -r requirements.txt") 安装依赖 安装成功
5Tool Call: terminal("curl localhost:8000/health") 验证端点 返回 {"status":"ok"}
6 文本回复 → 发给用户："项目已创建，FastAPI 在 8000 端口运行，/health 端点返回正常。"
🤔 注意观察：Agent 不是一次就搞定的。它一步一步来——先建目录、再写文件、再安装依赖、再验证——每步都是 LLM 看了上一步的结果后决定下一步。这就是 Agent Loop 的本质：感知→推理→行动→观察结果→再推理→再行动→……直到完成。

🔁 Agent Loop 流程图

📐 关键概念拆解

Turn（轮次）

1 次 LLM API 调用 = 1 个 Turn。如果 LLM 返回了 Tool Call，Agent 执行工具后把结果追加回 messages，这不算新 Turn——它只是当前 Turn 内的子步骤。

举例：上面 FastAPI 的例子中，6 步（5 次 Tool Call + 1 次文本回复）发生在 1 个 Turn 内。

并行 Tool Call

LLM 可以一次返回多个 tool call（如同时创建 3 个文件），Hermes 并行执行它们，而不是串行等待。

这大幅减少了 API 往返次数，显著提升效率。

Prompt Caching

System Prompt 在会话内不变，LLM Provider 可以缓存它。

后果：换工具、换技能、改配置后必须 /reset——改任何 System Prompt 内容都会破坏缓存，需要新会话重新加载。

Context Compression

对话历史太长时，Hermes 自动压缩旧消息（摘要化），确保不超出模型的 token 上限。

默认策略：上下文使用率超过 50% 时触发，压缩到 20%。

📌 本节要点
  • Agent Loop = Prompt → LLM → 工具执行 → 结果追加 → 循环
  • 一次用户消息可能触发多个 Turn，每个 Turn 内可能有多次工具调用
  • LLM 每次只决定 下一步做什么，不一次性规划全部
  • 防护机制：max_turns + Compression 防止失控
3 关键目录结构
所有数据默认存在 ~/.hermes/，可通过 $HERMES_HOME 环境变量自定义路径
~/.hermes/ ← 数据根目录
├── config.yaml ← 主配置文件（模型、Agent 参数、压缩策略…）
├── .env ← API Key 等密钥（不入 git，与 config 分离）
├── SOUL.md ← Agent 全局人格定义（始终加载）
├── skills/ ← 安装的技能文件（.md，按分类分目录存放）
├── state.db ← SQLite 会话数据库（FTS5 全文搜索）
├── sessions/ ← 会话路由索引 + 请求日志
├── logs/ ← Gateway 日志 + 错误日志
├── auth.json ← OAuth token + 凭证池
└── profiles/ ← 多 Profile 隔离目录
└── default/ ← 默认 profile（config.yaml + .env + skills）

📝 config.yaml

控制 Agent 全部行为的核心文件。包括：模型选择、Agent 参数（max_turns）、终端后端、压缩策略、Memory、委托配置等。

hermes config edit 直接编辑

🔐 .env

存放 API Key、Token。和 config.yaml 分离是为了密钥不入 git——config 可以分享，.env 绝对不能。

hermes config env-path 查看路径

🗄️ state.db

SQLite 数据库，存储所有会话的完整历史消息。内置 FTS5 全文搜索引擎——Agent 可以搜索任意历史对话。

这也是 session_search 工具的数据来源。

👤 profiles/

多套独立配置的隔离目录。每个 profile 有自己的 config.yaml、.env、skills、memory——就像一个独立的人格分身。

比如：一个 profile 连公司 GitLab，另一个连个人 GitHub。

💡 实用技巧：如果你想让两个项目使用完全不同的 Agent 配置（不同模型、不同工具、不同记忆），创建两个 Profile 比手动切配置方便得多。hermes --profile work vs hermes --profile personal。
4 System Prompt 是怎么拼出来的？
每次新会话启动时，Hermes 把所有"背景知识"拼接成一个巨大的系统提示词发给 LLM

理解 System Prompt 的拼接逻辑非常重要——因为Agent 的"认知边界"完全由它决定。Agent 能看到什么、知道什么、能做什么，都在这里。

🧱 8 层叠加结构

Layer 1 · Agent Persona（角色定义） "你叫 Hermes，是一个万能的机器人助手…"
Layer 2 · SOUL.md（全局身份文件） $HERMES_HOME/SOUL.md — 独立于项目，始终加载
Layer 3 · User Profile（用户画像） Memory: 姓名、偏好、环境设置…
Layer 4 · Agent Memory（Agent 自己的笔记） Memory: 项目路径、工具版本、经验教训…
Layer 5 · Active Skills（加载的技能文档） 当前会话中 /skill name 或 --skills 加载的 SKILL.md
Layer 6 · Project Context（项目上下文） .hermes.md / AGENTS.md / CLAUDE.md / .cursorrules （先到先得）
Layer 7 · Environment Hints（环境信息） OS: Linux x86_64 · Shell: bash · CWD: ~/projects
Layer 8 · Available Tools + Tool-Use Enforcement 所有可用工具的 JSON Schema + "你必须用工具做事，不能只描述"
💡 加载优先级（越靠上越核心）：
Persona > SOUL.md > Memory > Skills > Project Context

这意味着：Persona 里的指令优先级最高，不会被 Skills 或 Project Context 覆盖。如果你在 Persona 里写了"不要说谢谢"，即使 Skill 里建议说谢谢，Agent 也不会说。

📁 Project Context 文件的发现规则

文件发现方式适用场景
.hermes.md / HERMES.md 从当前目录向上遍历到 git root Hermes 专属，支持根目录 + 子目录层级继承
AGENTS.md / agents.md 仅当前工作目录 跨 Agent 可移植（Claude Code / Codex 都认）
CLAUDE.md 仅当前目录 Claude Code 风格，与 AGENTS.md 同类
.cursorrules 仅当前目录 从 Cursor IDE 迁移时使用

规则：先到先得——按上表优先级找到第一个就停止，不叠加。单个文件上限 20,000 字符，超出部分自动截断（保留头尾）。

📌 本节要点
  • System Prompt = 8 层内容叠加，决定了 Agent 的"认知世界"
  • 会话内不变 → 支持 Prompt Caching → 换任何内容都要 /reset
  • Project Context 先到先得不叠加——.hermes.md 和 AGENTS.md 只加载一个
  • Persona 优先级最高，是 Agent 行为的最底层约束
5 一次请求的完整数据流
假设用户通过飞书发送一条消息，追踪它的全链路——从平台适配到 LLM 再到返回

🔀 两条路径的对比

🔴 Tool Call 路径

LLM 认为信息不足、需要外部操作时走这条路径。

  • Agent 执行工具（终端 / 文件 / 浏览器…）
  • 工具输出追加到 messages
  • 再次调用 LLM（带工具结果）
  • LLM 重新判断：继续 Tool Call 还是给文本回复
  • 循环，上限 max_turns=90

🟢 文本回复路径

LLM 认为信息足够、可以直接回答时走这条路径。

  • LLM 返回纯文本（不调用工具）
  • Agent 结束当前 Turn
  • 文本经由平台适配器格式化
  • 投递到用户所在平台（飞书/Telegram/…）
  • ✅ 等待用户下一条消息
📌 本节要点
  • 平台适配器（Gateway）负责消息收发，Agent Loop 负责"思考+行动"
  • LLM 是关键决策者——每次返回 Tool Call 还是文本，由 LLM 决定
  • Tool Call 路径可能循环多次，直到 LLM 满意为止
  • state.db 是会话持久化的核心——适配器和 Agent Loop 都读写它

7 常见疑问
初学时最容易被问到的问题
Q: Hermes 和 LangChain / AutoGPT 有什么区别？
LangChain 是一个开发框架——你用它来构建 Agent。Hermes 是成品 Agent——你直接用它干活。
AutoGPT 是早期的自主 Agent 实验项目，已基本不维护。Hermes 更工程化、更稳定、生态更完整。
Q: Agent Loop 会不会死循环？
不会。有三层保护：
① max_turns=90——到上限强制终止
② Context Compression——token 快满时自动压缩，不完全依赖 turn 计数
③ 模型本身有判断力——当它觉得"可以回答了"就会返回文本，不走 Tool Call
Q: state.db 存了所有对话，隐私怎么办？
数据完全在本地，不上传任何第三方服务器。你可以随时删除：
hermes sessions prune --older-than 30 删除 30 天前的会话
也可以配置 privacy.redact_pii: true 在发给 LLM 前自动脱敏用户信息。
Q: 我能用本地模型（Ollama / llama.cpp）跑 Hermes 吗？
可以。Hermes 支持任何 OpenAI 兼容的 API 端点。配置 model.base_url 指向本地服务即可。
不过建议用足够强的模型（70B+ 或等价的），否则工具调用的准确度会下降。
Q: 我能同时用多个 LLM Provider 吗？
可以。Hermes 支持 Credential Pool——配置多个同 Provider 的 API Key，自动轮转避免限流。
不同 Provider 之间可以用 hermes model 随时切换，改完 /reset 生效。
